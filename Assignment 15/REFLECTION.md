## Lessons Learned
- **Communication is key**: Commenting on issues before submitting PRs helps avoid duplicate work and shows respect for the project owner.
- **Small, focused PRs** are much more likely to be reviewed and accepted than large changes.
- Following a project's existing style (especially documentation standards like Swagger) makes contributions smoother.
- Testing is crucial — running the full test suite before submitting gave me confidence in the changes.

## Challenges Faced
- Peer repositories vary greatly in maturity and documentation quality. Some had clear CONTRIBUTING.md files, while others required more exploration.
- Waiting for reviews can take time — collaboration in an academic setting depends on classmates' availability.
- Adapting to different tech stacks (Markdown-only, Spring Boot/Java, etc.) quickly.

## What Went Well
- Successfully identified three diverse projects and submitted quality PRs quickly.
- All changes were non-breaking and added clear value (especially documentation improvements).
- Gained real experience with forking, branching, and GitHub PR workflow.

## Improvements for Future Contributions
- Start earlier to allow more time for reviews and iterations.
- Reach out directly (politely) if a PR sits for too long.
- Aim for at least one feature addition where possible for bonus marks.

Overall, this assignment was excellent preparation for real open-source contribution. Even without merges yet, the process built valuable skills in collaboration and code review etiquette.
